# 提示工程

## 核心理念

提示工程不是"写更好的 prompt"，而是**设计任务上下文让模型输出符合预期**。技巧服务于可靠性，不服务于炫技。

## 实用技巧矩阵

| 技巧 | 何时用 | 示例 |
|------|--------|------|
| **角色设定** | 需要特定专业视角 | "你是一个 10 年经验的 Go 后端工程师" |
| **结构化输出** | 需要可解析结果 | "用 JSON 返回，字段：name, reason, confidence" |
| **Few-shot** | 输出格式难描述 | 给 2-3 个输入→输出范例 |
| **CoT (思维链)** | 多步推理 | "一步步分析，先列已知条件，再推导" |
| **自我修正** | 提高准确率 | "先给出答案，再检查一遍，如果不对就修正" |

## 结构模板

```markdown
## 角色
[定义 AI 的身份和专业领域]

## 任务
[一句话说清要做什么]

## 约束
- [硬性规则 1]
- [硬性规则 2]

## 输出格式
[期望的格式要求]

## 示例（可选）
输入: xxx
输出: xxx
```

## 常见反模式

| 反模式 | 问题 | 改进 |
|--------|------|------|
| prompt 过长 | 关键指令被稀释 | 精简到必要信息 |
| 指令模糊 | 模型随意发挥 | "不要做 X"比"请小心"有效 |
| 负面指令 | 反而强化错误行为 | "输出 JSON"比"不要输出 Markdown"有效 |
| 单次依赖 | 复杂任务一次搞不定 | 拆成多步，每步验证 |

## 编程场景模板

```
你是一个 [语言] 专家。
任务：实现 [功能描述]。
要求：
1. 包含错误处理
2. 添加类型注解/文档注释
3. 时间复杂度 [要求]
4. 只输出代码，不要解释
```
