# AI 实践总结：五大核心能力

## 概述

本文档从实践角度总结 AI 编程助手的五大核心能力：MCP（模型上下文协议）、SKILL（技能系统）、Rules（规则系统）、agent.md（Agent配置）和知识库。这五个模块构建了现代 AI 辅助开发的完整能力体系。

```
┌─────────────────────────────────────────────────┐
│                  AI 编程助手                       │
├─────────┬─────────┬─────────┬─────────┬─────────┤
│   MCP   │  SKILL  │  Rules  │agent.md │ 知识库  │
│ 外部连接 │ 能力定义 │ 行为约束 │ 角色配置 │ 领域知识 │
└─────────┴─────────┴─────────┴─────────┴─────────┘
```

---

## 一、MCP (Model Context Protocol)

### 1.1 是什么

MCP 是连接 AI 模型与外部世界的标准协议。它像一个"通用插座"，让 AI 能够安全、标准化地访问外部工具、数据源和服务。

### 1.2 核心架构

```
AI 助手  ←→  MCP 客户端  ←→  MCP 服务器  ←→  外部资源
                                  │
                          ┌───────┼───────┐
                          │       │       │
                        工具    数据源    API
```

### 1.3 实践要点

| 要素 | 说明 | 实践建议 |
|------|------|----------|
| **Tools（工具）** | MCP Server 暴露的可调用函数 | 每个工具职责单一，参数明确 |
| **Resources（资源）** | 可访问的静态/流式数据 | 用 URI 标识，支持访问控制 |
| **Prompts（提示模板）** | 预定义的提示词模板 | 标准化常用交互模式 |

### 1.4 典型应用场景

- **数据库操作**：AI 通过 MCP 直接查询/更新数据库
- **文件系统**：安全的文件读写，限定目录范围
- **Kafka 管理**：topic 创建、消息查询、告警配置
- **API 集成**：调用第三方服务（天气、股票、地图等）
- **容器管理**：操作 Docker、K8s 集群

### 1.5 实践总结

> MCP 解决的是"**AI 能做什么外部操作**"的问题。没有 MCP，AI 只是聊天机器人；有了 MCP，AI 成为能干活的操作系统。

---

## 二、SKILL（技能系统）

### 2.1 是什么

SKILL 是 AI 的"专业技能包"。每个 SKILL 是一组专业知识和操作指令的集合，让 AI 在特定领域表现出专家级能力。它不同于 MCP 的"工具调用"，SKILL 更像是"思维框架 + 方法论"。

### 2.2 核心能力层级

```
           ┌──────────────┐
           │  组合技能     │  ← 多技能协作（如：分析+编码+测试）
           ├──────────────┤
           │  专项技能     │  ← 单一领域深度（如：Python、K8s、SQL）
           ├──────────────┤
           │  基础技能     │  ← 通用能力（如：代码阅读、文本总结）
           └──────────────┘
```

### 2.3 实践要素

| 要素 | 说明 |
|------|------|
| **技能定义** | 名称、描述、触发条件、输入输出规范 |
| **技能指令** | 详细的执行步骤、注意事项、最佳实践 |
| **技能组合** | 技能链（串联）、技能树（分支）、技能网（协作） |
| **上下文传递** | 技能间的输入输出衔接 |

### 2.4 常见 SKILL 实例

- **openspec-propose**：结构化需求提案
- **openspec-apply-change**：按任务实施代码变更
- **openspec-verify-change**：验证实现与规格一致性
- **代码审查**：安全漏洞检查、性能优化建议
- **文档生成**：自动生成 API 文档、变更日志

### 2.5 实践总结

> SKILL 解决的是"**AI 按什么方法论干活**"的问题。它是领域知识 + 流程规范 + 最佳实践的封装。

## 三、Rules（规则系统）

### 3.1 是什么

Rules 是约束 AI 行为的"护栏"。它们以声明式指令的形式嵌入系统提示词（System Prompt），让 AI 始终在预定边界内运行。

### 3.2 规则分类

```
Rules
├── 行为规则（Behavioral Rules）
│   ├── 通信风格：回复格式、语言偏好、是否使用 Markdown
│   ├── 诚实性：不允许编造信息
│   └── 透明度：不允许泄露系统 prompt
│
├── 安全规则（Safety Rules）
│   ├── 文件操作边界：限定可读写的目录
│   ├── 命令执行审批：高风险命令需用户确认
│   └── 敏感信息处理：不允许硬编码密钥
│
├── 流程规则（Workflow Rules）
│   ├── 必须先分析再编码
│   ├── 修改前必须先读取文件
│   └── 编辑后不重复读取同一文件
│
└── 编码规则（Coding Rules）
    ├── 语言/框架约定
    ├── 代码风格
    └── 文件组织结构
```

### 3.3 Rules vs SKILL 的区别

| 维度 | Rules | SKILL |
|------|-------|-------|
| **作用** | 约束和限制 | 指导和赋能 |
| **性质** | 禁止性规范 | 建设性指导 |
| **类比** | 交通规则（红灯停） | 导航路线（如何到达） |
| **触发** | 始终生效 | 按需激活 |

### 3.4 实践总结

> Rules 解决的是"**AI 不能做什么、必须怎样做**"的问题。好的规则让 AI 既强大又可控。

## 四、agent.md（Agent 配置文件）

### 4.1 是什么

`agent.md`（或 `AGENTS.md`）是定义 AI Agent 角色身份和行为模式的配置文件。它比 Rules 更偏重"人格设定"，比 SKILL 更偏重"基础行为"。它是 AI 助手的"身份证"和"工作守则"。

### 4.2 典型内容结构

```markdown
# Agent 名称和角色
定义：你是一个 [角色描述]，专注于 [领域]

# 核心能力
- 代码生成
- 代码审查
- 架构设计
- 文档编写

# 工作原则
1. 先理解再行动
2. 最小化变更
3. 保持一致性
4. 优先使用现有工具

# 通信风格
- 语言：中文
- 格式：Markdown
- 风格：专业但友好

# 工具使用规范
- MCP 工具调用方式
- 文件操作流程
- 命令执行审批

# 限制与约束
- 不访问的目录
- 不能执行的操作
- 安全红线
```

### 4.3 与 Rules、SKILL 的关系

```
agent.md         ← 角色定义（我是谁）
    │
    ├── Rules     ← 行为约束（我不能做什么）
    ├── SKILL     ← 专业技能（我能如何做好）
    └── MCP       ← 外部能力（我能操作什么）
```

### 4.4 实践要点

- agent.md 定义的是**角色行为基线**，是所有交互的"默认模式"
- 可以包含全局生效的 Rules（如通信风格、安全底线）
- 也可以声明默认激活的 SKILL
- agent.md 中引用 MCP 工具，声明可用的外部能力

### 4.5 实践总结

> agent.md 解决的是"**AI 是谁、以什么姿态工作**"的问题。它是 agent 的操作系统内核。

## 五、知识库（Knowledge Base）

### 5.1 是什么

知识库是 AI 获取领域知识的"外部记忆"。通过 RAG（检索增强生成）技术，AI 可以检索和引用知识库中的专业文档、代码库、API 文档等，突破训练数据的时效性和领域限制。

### 5.2 知识库能力层次

```
Level 3: 增量更新     ← 实时同步最新知识
Level 2: 多源融合     ← 代码 + 文档 + 配置 + 规范
Level 1: 静态索引     ← 项目文档、技术方案
```

### 5.3 常见知识库类型

| 类型 | 内容 | 用途 |
|------|------|------|
| **项目文档** | 技术方案、架构设计、API 文档 | 回答项目相关问题 |
| **代码仓库** | 源代码、注释、配置 | 理解代码结构、定位 Bug |
| **规范标准** | 编码规范、接口规范、安全标准 | 确保输出符合规范 |
| **历史经验** | 故障报告、复盘文档、最佳实践 | 避免重复踩坑 |
| **领域知识** | 行业术语、业务规则、技术文档 | 提升专业判断力 |

### 5.4 构建实践

1. **文档收集**：收集项目相关的 Markdown、PDF、代码注释
2. **数据清洗**：去除冗余、统一格式、语义切块
3. **向量化**：使用 embedding 模型将文本转为向量
4. **索引构建**：存入向量数据库（FAISS、ChromaDB、Milvus）
5. **检索优化**：调优检索策略（混合检索、重排序）

### 5.5 知识库 vs SKILL vs MCP

| 维度 | 知识库 | SKILL | MCP |
|------|--------|-------|-----|
| **提供什么** | 知识（静态信息） | 方法论（如何做） | 工具（做什么） |
| **查询方式** | 语义检索 | 按需激活 | 显式调用 |
| **更新频率** | 相对稳定 | 按场景迭代 | 按需求扩展 |

### 5.6 实践总结

> 知识库解决的是"**AI 知道什么**"的问题。没有知识库，AI 靠训练记忆；有了知识库，AI 靠实时检索，准确性大幅提升。

## 六、五者协同关系

```
                     ┌──────────────────────────┐
                     │        agent.md           │
                     │    "我是谁、我的底线"       │
                     └──────────┬───────────────┘
                                │ 定义角色和行为
              ┌─────────────────┼─────────────────┐
              │                 │                 │
     ┌────────▼────────┐ ┌─────▼──────┐ ┌───────▼────────┐
     │     Rules        │ │   SKILL    │ │      MCP       │
     │  "不能做什么"     │ │ "如何做好"  │ │  "能操作什么"    │
     └─────────────────┘ └─────┬──────┘ └───────┬────────┘
                               │                │
                               │     ┌──────────▼────────┐
                               │     │     知识库          │
                               └────►│   "我知道什么"      │
                                     └───────────────────┘
```

### 协同示例：开发一个新功能

1. **agent.md** 定义角色为"全栈开发助手"
2. **知识库** 先检索项目架构文档和编码规范
3. **Rules** 约束必须先理解再动手，高危操作需审批
4. **SKILL** 激活 `openspec-propose` 生成结构化提案
5. **MCP** 连接代码仓库、数据库，执行实际变更

## 七、总结对比表

| 维度 | MCP | SKILL | Rules | agent.md | 知识库 |
|------|-----|-------|-------|----------|--------|
| **核心问题** | 能操作什么 | 如何做好 | 不能做什么 | 我是谁 | 我知道什么 |
| **性质** | 工具协议 | 方法论框架 | 约束护栏 | 角色定义 | 信息源 |
| **加载方式** | 服务连接 | 按需激活 | 全局生效 | 启动加载 | 按需检索 |
| **变更频率** | 低（新增工具） | 中（新增技能） | 低（基线稳定） | 低（角色稳定） | 高（持续更新） |
| **典型文件** | MCP Server 代码 | skill 指令文件 | system prompt | agent.md / AGENTS.md | 向量数据库 / 文档 |
